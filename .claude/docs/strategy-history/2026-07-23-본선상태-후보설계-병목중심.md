# 본선 상태·후보 설계 — 병목중심 관측 (2026-07-23, YR-088 파생)

> 사용자 피드백 기반. 우리는 **크레인에 액션**을 하므로 본선을 챙기려면 "본선 작업인가?"가
> 아니라 **"이 크레인 후보가 지금 본선 병목을 얼마나 푸는가?"** 가 중심이어야 한다.
> 코드 검증 완료 — 아래 진단 3건 모두 사실.

## 1. 진단 (코드 대조 — 다 맞음)

- **현재 본선 상태 = 8필드**(slack_s·risk·delay_symptom·remaining_moves·remaining_service_time·
  sts_wait·transfer_wait·expected_delay) [schema.py:211].
- **slack 이 STS 평균간격만 반영** — `남은작업시간 = 남은move × STS간격`, YC 대기·재조작·YT 이송·
  버퍼 부족·장비경합 **미반영** [vessel.py:53]. → 명목 여유일 뿐 실질 마감 여유 아님.
- **양하 역압력 공백** — 이송 도착 순간 버퍼 비우고 YC 작업 해제 → YC 가 늦게 적재해도 STS 에
  역압력 안 감 [engine.py:779]. 적하는 반대(YC 늦으면 버퍼 비어 STS 막힘) → **역압력 있음**.

## 2. 결정 — 본선 관측을 "스케줄중심 **최소**"로 (2026-07-24 사용자 최소화 반영·검증)

★정정: 앞선 병목중심 12필드 리스트는 STS/YT 내부를 과노출했다. **야드크레인 정책이 조종 못 하는
STS·YT 작업/유휴 raw 는 정책에 넣지 않는다**(상태공간만 키우고 학습·일반화 방해). 시뮬은 전부
모델하되 **정책엔 결정에 필요한 요약만.**

**정책 상태(YC)** = 컨테이너 + 본선 스케줄 급함 + 흐름여유 1개:
- **트럭 컨테이너 상태** — 누적대기·SLA·blocker·위치 (기존).
- **본선 컨테이너 상태** — 어느 양적하 작업이 지금/곧 필요·위치·blocker.
- **schedule_slack**(feedforward) — 알려진 스케줄로 계산한 마감 대비 급함 = **트럭 ETA 의 본선판**.
  이걸로 "본선 가중치 올릴까" 결정. **가장 급한 선박(min slack) 보호**(평균 아님).
- **flow_margin**(feedback) — STS 굶기(적하)/막히기(양하)까지 여유 **한 숫자**. YT/버퍼를 raw 노출
  없이 요약 — 스케줄만으론 못 잡는 경합·교란을 잡음.

**정책 상태 아님(시뮬만)**: STS/YT 작업·유휴·버퍼 raw·이송 대기 세부.

**후보별 본선효과 (per-candidate, 최소)**: `linked_vessel` · `is_next_needed_move`(스케줄상 다음 필요
move 인가) · `slack_gain_s`(이 작업이 마감 여유를 몇 초 당김). 기계 노출형(unblocks_sts raw) 대신
스케줄·여유 기준으로 "이 작업이 배를 돕나"를 후보에.

**구현·검증** [vessel_signals.py](../../src/yard_rl/integrated/vessel_signals.py) · tests 5:
schedule_slack 마감 다가올수록 감소 · flow_margin STS 막힌 순간 **100% ≤0**(동행 정확)+막히기 전
선행 · 최소상태 **요약 2개만**(STS/YT raw 누출 0) · 가장급한=min slack.

## 3. 핵심 파생값 (구현됨 — vessel_signals.py)

```
schedule_slack (feedforward) = 계획 선석종료 − 현재 − 남은move × cadence   (음수=계획상 초과 위험)
flow_margin    (feedback)    = 버퍼가 현 STS 속도를 버틸 시간 − 다음 YT 공급/배수 예상
  (적하: 버퍼 비면 STS 정지 / 양하: 버퍼 차면 STS 정지 — 부호 반대를 한 값으로 통일)
  음수면 아직 안 멈췄어도 곧 멈출 위험.
```
정정: 앞서 "critical_slack = 예상 잔여 중요경로시간(YC/YT/버퍼 전부)"은 정확값이 사실상
rollout(§5)이라, 정책엔 **feedforward(schedule_slack) + feedback(flow_margin)** 싼 근사만 넣고
정확한 중요경로는 위험할 때 rollout 몫으로 둔다(과모델 회피·상태공간 최소).

## 4. ★선결 — 양하 역압력 엔진 수정 ✅ 완료 (2026-07-23, `c93918e`)

양하는 이송 도착 순간 quay 버퍼 해제·YC 완료 시 무동작이라 **YC 적재 지연이 STS 에 역압력 안
감**(양하 결정권 공백)이었다. **수정**: `TerminalSimulator(yard_handover_cap=…)` opt-in
(기본 None=현행 **골든 바이트 동일**). ON 이면 STS-뽑기~YC-적재 사이 미완 양하 박스(인계
파이프라인)를 cap 으로 제한 → **YC 가 늦으면 파이프라인 만재 → STS 정지 → 배 완료 지연**.
VesselProcess 스키마 불변(sim 레벨 dict, 골든 안전). **검증**: 골든·스냅샷 5 + 회귀 131 통과
(OFF 불변) · 역압력 테스트 4(0<peak≤cap·cap 조일수록 STS 대기↑ 단조). → 이제 양하
critical_slack·unblocks_sts 가 **정책이 실제로 바꿀 수 있는 신호**다. 남은 정합: cap 값(assumed)
근거·본선 상태 feature 가 이 파이프라인을 참조.

## 5. 정직한 단서 (적대검토 규율 반영)

- **critical_slack·marginal_slack_gain 의 정확 계산 = YC/YT/버퍼 미래 예측 = 사실상 짧은 rollout.**
  → feature 는 **싼 근사**로 두고 정확 판정은 rollout 이. feature 가 rollout 인 척 금지.
- 이 상태가 **관측별칭을 실제로 줄이는지 측정 선결**(YR-088 1번 게이트) — 그냥 넣는다고 되는 것
  아님(YR-087: 246필드 넣어도 별칭으로 실패). 라벨 충돌률로 값/랭킹 타깃에서 확인.
- **rollout(계획법)은 이미 이 정보를 전체 시뮬로 암묵 사용** → 새 상태의 1차 수혜자는 (a) 학습
  가치망(B) (b) 위험시만 rollout 켜는 발동 트리거. rollout 자체엔 필수 아님.

## 6. 정책 설계 결정 (종합)

- **관측** = per-vessel 병목상태(§2A) + **per-candidate 병목해소 효과(§2B, 1차)**. 후보가 본선
  컨테이너를 참조해 "이 크레인 행동이 배 병목을 푸나"를 표현.
- **가장 위험한 배 보호**(평균 아님).
- **행동**은 그대로 통일 2작업(STORE/RETRIEVE) — 새 행동 없음. 신호는 상태·후보 feature 로.
- **순서**: ① 양하 역압력 엔진 수정(조율) → ② critical_slack + 후보 병목효과 feature(싼 근사)
  구현 → ③ 별칭 감소 측정(B 게이트) → ④ 통과 시 가치망 재학습 또는 발동 트리거.
- 범위: 문헌 보정 시뮬 조건 · 실운영 주장 금지. 미확보(YC/YT 실측 시간분포)는 assumed.

## 7. 한 줄

본선 관측의 중심을 **"위험도 1개"에서 "실질 마감여유 + 병목위치 + 후보의 병목해소 효과"** 로
바꾼다. 단 (a) 양하 역압력 엔진 수정이 선결이고, (b) 정확 slack 은 rollout 몫·feature 는 근사이며,
(c) 별칭 감소를 먼저 측정해야 이 상태가 학습에 값을 한다.
