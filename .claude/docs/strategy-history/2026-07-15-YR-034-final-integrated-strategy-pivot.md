> 기준 문서: [부산항 레인–다중 YC 협조 최적화 최종전략](../부산항_레인_다중야드크레인_협조최적화_강화학습_최종전략.md)

## 1. 결정

앞으로의 제품 목표는 단일 YC의 트럭 평균대기 최소화가 아니라 다음이다.

> 사전 반출입정보·ETA·컨테이너·본선·이송장비·레인·다중 YC 상태를 함께 보고,
> 실행 가능한 다음 작업을 협조적으로 골라 터미널의 정규화 누적 운영비용을 최소화한다.

최종 정책은 하나만 둔다. 과거 Exp-1~4처럼 기능마다 별도 정책을 만들지 않고, 동일한
통합 정책에서 ETA·선제정리·본선위험·레인·다중 YC 요소를 제거하는 ablation으로 각
기능의 효과를 측정한다.

## 2. 구현 계약

| 영역 | 동결한 방향 |
|---|---|
| Q 목표 | `Q_cost(global, crane, candidate)` = 후보 선택 후 예상 할인 누적 터미널 비용 |
| 선택 | 실행 가능한 후보 중 `argmin Q_cost`; 안전·물리·마감 위반은 비용이 아니라 mask |
| 후보 | 외부트럭·본선·재조작·위치조정·제한적 양보/대기의 가변 후보 |
| 로컬 학습기 | Candidate DQN → **Double DQN 기본** → Dueling 추가효과 비교 |
| 협조 | 중앙 joint-action resolver 기준선 후, 검증된 로컬 utility를 QMIX mixer에 연결 |
| 실행 | 공동 제약이 있으므로 운영 배정은 중앙 resolver가 최종 승인 |
| 시간 | 이벤트 기반 SMDP와 실제 경과시간 할인, ETA·deadline·Slack 잔여시간 입력 |
| 장치 | 시뮬레이션은 CPU, batch 학습은 GPU 권장·CPU fallback, 운영 추론은 CPU 기본 |

“처음부터 통합”은 모든 최종 도메인과 tensor 필드를 첫 계약에 넣는다는 뜻이다. 서로
검증되지 않은 시뮬레이터·로컬 Q망·mixer를 한 번에 학습한다는 뜻은 아니다. 실행 가능한
수직절편은 계약→환경→비용→로컬 평가→공동배정 순으로 검증하되, 별도 Exp 정책으로
갈라지지 않는다.

## 3. 결정 경위와 보존할 증거

| 시점 | 증거 | 최종전략에 준 영향 |
|---|---|---|
| YR-027 | 최소상태로 coverage를 해결해도 순수 Q가 shortest-service보다 +1.195분 | Q-table 확대보다 순서품질과 함수근사가 핵심 |
| YR-028 | fallback 55% 직접원인은 checkpoint 규칙, fallback 감소와 성능악화가 단조 | coverage와 정책품질을 분리하고 선택 프로토콜을 별도 관리 |
| YR-030-b/c | greedy prior·할인·입도·tabular 잔차를 소진, 최고도 greedy 미달 | 후보 직접평가 골격은 유지하되 표를 신경망으로 교체 |
| YR-012 | 연속 Δ-net이 격차를 +0.083분까지 줄이고 P95를 최초 개선 | bucket 해상도 손실이 실재, 연속 feature 필요 |
| YR-031 | beam oracle 상금 하한 +0.182분, 90/100일 개선 | greedy를 넘을 여지가 실제로 존재 |
| YR-012-b | replay+target은 진동만 줄이고 성능 악화 | 안정된 곡선이 채택 근거가 아니며 locked 성능으로 판단 |
| YR-031-b | H-A AUC 0.852 지지, H-B 집합이득 0.000 기각 | 대기열 요약 feature는 필요하나 후보 독립 점수 골격은 충분 |

이 수치는 합성 시나리오·가정 프로파일의 단일 YC 증거다. 최종 통합정책의 성능이나
부산항 실운영 개선을 증명하지 않으며, 아키텍처 선택과 회귀 기준으로만 사용한다.

## 4. 검토한 대안과 기각 사유

| 대안 | 기각 사유 |
|---|---|
| 트럭-only Δ-net 튜닝을 최종제품으로 계속 | 본선·내부이송·레인 비용을 보지 못해 자원배분 목표가 잘못될 수 있음 |
| 최종 State를 Q-table로 이산화 | 가변 후보와 연속 Slack·ETA·다중 YC 조합을 표로 일반화하기 어려움 |
| 전체 joint action을 단일 고정 DQN head로 출력 | YC 수·후보 수에 따라 조합이 폭발하고 Job ID 일반화가 불가능 |
| QMIX만 먼저 붙여 원인을 한꺼번에 학습 | 환경·비용·로컬 utility·mixer 실패를 분리할 수 없음 |
| 각 YC가 완전히 독립 실행 | 동일 Job·레인·비통과 제약 때문에 독립 argmin이 실행 불가능할 수 있음 |
| ETA용·본선용 정책을 각각 운영 | 정책 간 목표와 비용이 갈라져 같은 자원을 일관되게 배분할 수 없음 |

## 5. Dashboard 전환

- YR-031-b를 결과와 함께 Done으로 이동하고, 파생된 고정 대기열 요약 검증을 YR-012-c로 등록한다.
- 중복 사용된 checkpoint 작업 ID를 YR-032에서 YR-033으로 바로잡는다.
- 기존 미래정보 전용 YR-032는 별도 정책 계획을 폐기하고 YR-014 통합 ablation에 흡수한다.
- YR-026의 트럭-only 비용 민감도는 YR-038 통합 Total Cost 계약으로 흡수한다.
- YR-035~039를 통합 계약·시뮬레이터·후보·비용·로컬 Q망 work package로 등록한다.
- YR-013은 “Exp-4”가 아니라 중앙 공동배정·QMIX 협조 계층으로, YR-014는 동일 최종
  정책의 locked 평가·ablation으로 재정의한다.

## 6. 적용 범위와 재검토 조건

- YR-002 실자료와 YR-009 validation 전에는 실운영 효과를 주장하지 않는다.
- 중앙 matching이 모든 locked 조건에서 QMIX와 동등하거나 우수하면 mixer를 운영에
  채택하지 않는다. 최종 구조의 후보라는 이유만으로 복잡도를 강제하지 않는다.
- Candidate Double DQN이 같은 정보·행동의 강한 휴리스틱을 넘지 못하면, QMIX 이전에
  feature·비용·후보 누락을 재감사한다.
- 실제 공동 서비스영역 자료가 독립 YC 구조를 보이면 협조 범위를 해당 물리구조에 맞게
  축소한다.
- 비용 가중치는 합성 결과로 확정하지 않고 실제 단위·운영 허용치와 민감도 분석으로 고정한다.
