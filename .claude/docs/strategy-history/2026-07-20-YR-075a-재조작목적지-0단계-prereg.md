# YR-075-a 0단계 사전등록 — 재조작 목적지 오라클 천장 (2026-07-20)

> 실행 전 동결. 결과는 `## 실행 결과` 로 append. 사용자 승인 "이렇게 진행해줘".
> 설계 3단계(게이트형) 중 0단계 = **만들기 전에 상금이 있는지 먼저 잰다**
> (YR-070/072 규율). 헤드룸 ≈ 0 이면 "greedy 목적지 near-optimal" 박제·종료,
> 실재하면 1단계(목적지 후보 확장)·2단계(증류 통합) 진행.

## 1. 질문

방해 컨테이너를 치울 때 목적지를 정하는 현 고정 규칙(`find_slot` = 최근접·저더미
greedy, [stack.py:101])이 **미래를 아는 배치보다 유의하게 나쁜가.** 나쁘면 "어디에
놓을지"를 결정으로 열 가치(1·2단계)가 있고, 대등하면 greedy 가 이미 최적.

## 2. 오라클 (동결)

- 주입: 통합 엔진 `_plan` 의 재조작 목적지 선택([engine.py:394])에 opt-in 훅
  `slot_selector`. None(기본)이면 `find_slot` 그대로 = **골든 바이트 동일** (sim/
  stack.py 무수정 — 통합 레벨 주입).
- 오라클 = **미래 재조작 회피 배치**: 각 blocker 를, 후보 슬롯 중 (① 목적지 더미에
  이 blocker 보다 **먼저 반출**될 컨테이너 수 [= 새 미래 blocker 생성 위험] ②
  greedy 비용) 사전식 최소 슬롯에 놓는다. 미래 반출시각은 시나리오 jobs 에서
  전지적 추출(오라클 지위 — CTDE·평가 전용, 정책 아님). blocker 자신이 무예약이면
  반출시각 ∞ (예약된 컨테이너 위 적재를 최대 회피).

## 3. 조건·비교 (동결)

- 정책 고정 = **SF_SPT** (목적지 외 전 결정 동일 — 차이를 목적지로 격리).
- 지형: v2 × 고장치율 (재조작 밀도 확보 필수 — 기본 fill 0.30 은 재조작 희소).
  셀: fill {0.55, 0.70} × 부하 {mid, high}. seed **신규** 748000~748019 (수준·fill
  공유, 차이는 목적지 규칙뿐 — paired).
- arms: GREEDY(현 find_slot) · ORACLE(미래인지). paired bootstrap 10k(75168).

## 4. 판정 (동결)

- **헤드룸 지표**: paired Δtotal_cost(ORACLE−GREEDY) + **Δrehandle_count**(직접
  기전) + Δmean_wait. 절대차+CI 보고.
- **게이트**: 어느 셀이든 Δtotal CI 상한 < 0 (오라클 유의 개선) **또는** Δrehandle
  CI 상한 < 0 이면 **헤드룸 실재 → 1단계 진행 권고**. 전 셀에서 두 CI 모두 0 포함/
  양수면 **near-optimal 박제 → YR-075-a 종료**, 다음 축(전문가 반복 등)으로.
- guard: 완주 전 seed 1.0·backlog 0·결정론(오라클 2 seed 재실행 동일)·골든 불변
  (fill 0.30·selector None 시 기존 지문 불변 회귀).
- 의미: 문헌 보정 조건·오라클은 상한 근사(전지적 미래·per-blocker greedy — 진
  최적 아님, 헤드룸의 하한 추정). 실재 판정은 강함, 부재 판정은 "이 오라클로는
  못 찾음" 한정.
