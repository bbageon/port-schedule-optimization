# YR-051 — "미래정보가 성능을 나쁘게 하나?" 검증과 baseline 라이브락 수정

> 계기: 사용자 질문 "ETA(미래정보)를 넣었을 때 성능이 나빠진다는 게 상식적으로 말이 안 되는데?"
> 결론: **사용자 직관이 옳다.** 순수 정보는 악화시키지 않았고, 관찰된 악화는 baseline의 버그였다.

## 1. 원리 — 정보의 가치는 음수가 될 수 없다 (단, 조건부)

정보의 가치 정리(Value of Information ≥ 0)는 **최적 의사결정자**에게만 성립한다. 정보를 최적으로
쓰는 주체는 새 정보로 손해 볼 수 없다 — 최소한 무시하면 되기 때문. 그래서 "미래정보를 넣었더니
나빠졌다"가 사실이라면 둘 중 하나다: (a) 의사결정자가 최적이 아니거나, (b) 그건 "정보"가 아니라
정보에 딸려온 다른 변화다. 검증은 이 둘을 갈랐다.

## 2. 실측 — 순수 정보 vs 정보+새행동 (seed 8개)

ETA 주입은 두 가지를 동시에 한다: ① ETA를 **feature(정보)**로 노출, ② ETA가 있어야 열리는
**새 행동**(선제 재조작·미래 도착 bay로의 위치이동)을 후보로 추가. 이 둘을 분리했다.

| arm | ETA 정보 | 새 행동 | ServiceFirstSPT 평균대기 | 총비용 |
|---|---|---|---|---|
| A. ETA 없음 | ✗ | ✗ | 0.632분 | 109.18 |
| B. ETA 정보만 (새 행동 차단) | ✓ | ✗ | **0.632분 (A와 완전 동일)** | 109.18 |
| C. ETA 전체 | ✓ | ✓ | 0.673분 | 83.84 |

**핵심: A = B가 소수점까지 완전히 같다.** 고정 규칙은 ETA feature를 애초에 안 읽으므로 순수
정보의 효과가 정확히 0이다 — 악화가 아니라 무효(VoI = 0, 정리와 일치). 악화처럼 보였던 것은 전부
arm C의 **새 행동** 쪽에서 왔다.

## 3. 그 "새 행동 악화"의 정체 — baseline 라이브락 (실제 버그)

새 행동이 나쁜 게 아니라, `JointRolloutGreedy`(YR-044 1차 baseline)가 새 행동 앞에서 멈춰버렸다.

- **증상**: seed 8개 중 4개(310003~310006)에서 완료율 0%, 대기 240분, 아무 작업도 안 함
  (SERVE·REPOSITION·PRE 전부 0회 = WAIT 무한반복).
- **근원**: `_combos`가 후보 조합이 max_combos(64)를 넘으면 각 크레인을 앞 8개로 자르는데, WAIT는
  후보 정렬상 맨 끝이라 **잘려나간다**. ETA가 후보를 12개로 늘려 절단이 자주 발동 → 남은 "둘 다
  실작업" 64조합이 서로 토큰/코리도 충돌로 전부 공동 실행불가 → decide가 진행 가능한 조합을 하나도
  못 찾고 합성 WAIT로 폴백 → 다음 결정도 동일 → 라이브락.
- **확정**: max_combos=900(절단 제거) 시 4개 seed 전부 240분→1분 내외·완료율 100%로 정상 →
  절단이 근원임을 못 박음.

**대조가 결정적이다**: 같은 seed·같은 ETA에서 고정 규칙(SF·FIFO)은 rollout을 안 쓰므로 전부 무사
(완료율 100%·대기 0.6분). 즉 정보가 나쁜 게 아니라 특정 baseline의 조합 열거 로직이 취약했다.
그리고 YR-044 건전성 계약(`serve_when_available`<0.25)이 4개 실패를 **전부 검출**했다 — 검출기는
제 역할을 했다.

## 4. 수정 (YR-051)

`_combos`의 절단이 **WAIT를 항상 보존**하도록 고쳤다 → 부분작업 조합(SERVE/WAIT, WAIT/SERVE)이
항상 열거되어 SERVE가 하나라도 가능하면 진행한다. 최소한의 feasible 조합(WAIT/WAIT)도 항상 확보.
`BeamLookahead`도 같은 `_combos`를 써서 함께 보호된다.

검증: 8 seed 전부 완료율 100% 복구, **seed 310000 총비용 62.927 불변**(정상 seed는 안 건드림).
라이브락 회귀 테스트가 310003~310006 완주를 강제한다.

## 5. 파급

- **YR-048 §5 정정**: `62.927 < 79.643`(seed 310000)은 하필 정상 seed의 값 — baseline 우위로
  인용 금지. 대표 비교는 YR-045 신규 seed 대역에서 재측정.
- **YR-045 선결조건 추가**: baseline이 신규 seed에서 라이브락 없이 완주함을 arm별로 재확인.
- **교훈**: seed 1개(310000)만 측정하면 절반이 실패하는 baseline을 "건전"으로 오인한다 —
  YR-039 무효판정의 "seed 다양성" 교훈과 같은 계열. 조합 절단 같은 조용한 축소는 특정 상태에서
  치명적일 수 있으므로 항상 no-op 안전판을 남긴다.
