# YR-050 — ETA 선제작업 결정 시점·연착 신호 계약

> YR-048 적대 리뷰 파생 2건의 해소. 코드 원본은 `src/yard_rl/integrated/engine.py`(wake·
> 결정 개방)·`candidates.py`(공유 술어)·`src/yard_rl/contract/schema.py`(itc-v2),
> 검증은 `tests/integrated/test_yr050_decision_epoch.py`(10건)다.

## 1. 문제와 결론 (쉬운 말)

두 가지가 막혀 있었다.

1. **결정이 안 열림**: 엔진은 "지금 처리할 트럭(SERVE)"이 있어야만 크레인에게 무엇을 할지
   묻는다. 트럭 도착예정시각(ETA)이 미리 보여도 한산한 시간대·첫 트럭 도착 전에는 질문
   자체가 없어, 선제 재조작(PRE_REHANDLE — 트럭이 오기 전에 반출 컨테이너 위 방해물을
   미리 치우는 작업)을 고를 수 없었다.
2. **연착 신호가 지워짐**: ETA는 지났는데 트럭이 아직 안 온 구간(오차 최대 300초)에서
   "도착까지 남은 시간" feature(`predicted_arrival_gap_s`)가 0 하한으로 잘려, "지금 도착
   예정"(0)과 "연착 중"(음수)이 학습 입력에서 같은 값이 됐다.

해소: (1) ETA가 결정 지평(1800초)에 들어오는 순간을 **wake**로 두고 그 시점에 결정을 연다.
(2) 0 하한 절단을 제거해 음수(=연착 정도)를 그대로 전달한다 — 계약 버전 **itc-v1 → itc-v2**.

## 2. 구현 계약

| 항목 | 동결 내용 |
|---|---|
| wake 시각 | `max(0, provided_eta − decision_horizon_s)` — GATE_OUT(반출)·ETA 보유 job 만 |
| wake 소비 | PRE_ADVICE 정보수준에서만. 1회성. event_log 에 `ETA_WAKE` 로 가시화 |
| 질문 예산 | **wake 1회당 크레인별 1회(armed)** — 결정에 한 번 포함되면 소진, 바쁜 크레인은 유휴화 시점까지 유지 |
| 결정 개방 | SERVE 실행가능 ∪ (armed ∧ 선제 재조작 기회 `eta_opportunity`) |
| 술어 공유 | 개방 술어와 후보 생성기가 같은 게이트 함수(`iter_pre_rehandle_jobs`)를 읽음 — 어긋남 구조적 차단 |
| 정보 경계 | `actual_*` 미열람. GATE_IN·BLOCK_ARRIVAL 은 wake 완전 비활성 → 이벤트 스트림·golden 불변 |
| 연착 신호 | `predicted_arrival_gap_s` 부호 보존(clip_lo 제거) — 음수 = ETA 경과·미도착 |
| 계약 버전 | itc-v2 (스키마 자체 규약 "필드·범위 변경 = 버전 bump + golden 재생성" 준수). 벡터 길이·필드 구성은 불변 — 학습 텐서 차원 동일 |

## 3. 폐기한 1차 설계 — 기회 잔존 동안 무제한 재질문

처음에는 "선제 기회가 남아있는 한 모든 이벤트가 결정을 재개방"으로 구현했다. 실측 결과
결정이 폭증하고 REPOSITION(빈 크레인 이동)이 장악했다 — WAIT를 최하위로 두는 baseline이
질문받을 때마다 "가장 짧은 행동"인 REPO를 고르고, 그 완료 이벤트가 다시 질문을 여는 되먹임.

| seed 310000, SF-SPT | 변경 전 | 1차(무제한) | 최종(armed) |
|---|---:|---:|---:|
| 결정 수 | 40 | 464 | **61** |
| REPOSITION 선택 | 13 | 410 (**88%**) | 33 (32%) |
| 총비용 | 79.643 | 113.228 | 89.922 |

REPO 88%는 YR-039 무효 사유였던 퇴화 패턴의 재현이고 행동 건전성 계약(비-SERVE 60% 상한,
YR-044)도 깨진다. 그래서 **정보 이벤트(wake)당 크레인별 1회 질문**으로 확정했다 — 선제
기회는 실행하면 방해물이 사라져 자연 소멸하지만, 위치선점(REPO) 기회는 계속 재발생하므로
개방 술어에서 REPO를 의도적으로 제외한 것도 같은 이유다(REPO 후보 자체는 열린 결정에서
기존대로 발행).

## 4. 발생 실측 (변경 전 → 후, PRE_ADVICE)

`ResolverPolicy(ServiceFirstSPT)`·`JointRolloutGreedy(600s)` — YR-048 §5 측정과 동일 조건.

| 정책·seed | 결정 수 | 행동분포 (PRE/REPO/SERVE/WAIT) | SERVE 선택률¹ | 완료율 | 총비용 |
|---|---|---|---|---|---|
| SF-SPT 310000 | 40 → 61 | 1/13/41/16 → 0/33/41/29 | 0.578 → 0.562 | 1.0 → 1.0 | 79.643 → 89.922 |
| SF-SPT 310007 | 54 → 88 | 0/28/54/13 → 0/67/54/17 | 0.568 → 0.575 | 1.0 → 1.0 | 96.700 → 91.885 |
| SF-SPT 700123 | 55 → 89 | 1/33/55/8 → 2/67/55/10 | 0.567 → 0.579 | 1.0 → 1.0 | 89.381 → 89.294 |
| JointRollout 310000 | 53 → 66 | 3/34/41/12 → 1/50/41/25 | 0.456 → 0.494 | 1.0 → 1.0 | 62.927 → 62.558 |

¹ SERVE 선택률 = 실작업이 가능했을 때 실제로 고른 비율(`serve_when_available`, 건전성 문턱 0.25).

해석: 추가 결정은 wake 수 × 크레인 수로 유계(+13~34건). 건전성 계약(0.25 하한·60% 상한)은
전 조건 통과. 1차 baseline(JointRollout)의 총비용·완료율은 동등(62.9→62.6). SF-SPT의 wake
질문 응답이 REPO로 쏠리는 성향(+10 비용, seed 310000)은 SPT 계열의 알려진 성질(최단 행동
선호)이며 진단군 역할에는 영향 없다. rollout 시간 +21%(9.0→10.9s/에피소드).

## 5. 검증 (전용 10건 + 회귀 — Windows 순수 파이썬 **297 passed**)

유일한 실패 1건은 YR-050 무관 — 신규 머신 Anaconda 의 구버전 streamlit 이 UI 신 API 를
몰라 나는 환경 비호환(`test_streamlit_app_renders`, YR-053 이 담당). torch 8파일은 이 머신
WSL venv 미구성으로 미실행(미실행은 통과가 아니라 미지 — YR-046 교훈, YR-053 후 재검증).

- 첫 실제 도착 전 PRE 결정·완료, 도착 SERVE 재조작 0 (혜택 경로) / ETA 없으면 도착 전
  결정 0 (counterfactual) / 바쁜 크레인 armed 유지 → 유휴화 시점 선제 결정.
- 누출 0: wake 는 `provided_eta`·지평만 사용 — 도착 진실을 바꿔도 도착 전 결정열 동일.
  GATE_IN·BLOCK_ARRIVAL 은 ETA 유무와 무관하게 이벤트 해시 동일.
- 전-WAIT 정책도 유한 결정으로 완주(같은 시각 재결정 없음)·결정론 재현.
- 음수 gap: 연착 구간 후보가 음수 값으로 `validate_all` 통과 record 에 실림 + 스키마
  단위검증(−120 보존·BLOCK_ARRIVAL known=0).
- golden: BLOCK_ARRIVAL 터미널 golden(n_events 115·hash `970b45a2…`) **무수정 통과** —
  낮은 정보수준 거동 불변의 직접 증거. 계약 golden 3종은 itc-v2 로 재생성했고, PRE_ADVICE
  record golden 은 **버전 문자열 외 완전 동일**(minimal fixture 에 ETA 없음 → 거동 불변).
  계약 fixture 에 연착(−180)·미도착(+600) PRE 후보를 추가해 부호 커버리지를 박제.

## 6. 영향·한계·후속

- **YR-045 3-arm 과의 관계**: NO_ETA arm 은 `provided_eta` 부재로 wake 0건(자연 비활성).
  ETA_NO_PRE arm 도 wake·결정 개방은 동작(개방 술어는 arm 의 행동 차단과 독립) — arm 간
  결정 기회 동등성이 유지된다. 사전등록 문서는 동결이므로 미수정.
- **RL 관점**: 학습 정책은 wake 결정에서 WAIT/PRE/REPO 를 자유 선택 — 조기 선제작업의
  가치 판정은 YR-045 의 몫이다. 여기서는 "결정이 존재하게" 만들었을 뿐이다.
- **한계**: wake 는 시나리오 고정 ETA 기준 1회 — 동적 ETA 갱신·미등록·편향(YR-019)은 범위
  밖. armed 소진 후 blocker 구성이 바뀌어도 다음 wake 까지 선제 전용 결정은 안 열린다
  (SERVE 주도 결정에서는 PRE 후보가 계속 발행되므로 정보는 유실되지 않음).
- 이 머신(신규 클론)의 WSL torch 환경 미구성 발견 — torch 8파일 검증은 환경 구성(YR-053)
  후. 본 변경은 스키마 벡터 길이 불변이라 학습 코드 차원 영향 없음.
