# 05. 테스트·로드맵·산출물

[← 상위 구현계획서](../../부산항_야드크레인_강화학습_구현계획서.md)

## 1. 테스트 전략

### 1.1 Unit Test

- 좌표·거리·이동시간 계산
- blocker 탐색과 합법적 목적 슬롯 탐색
- 정보수준별 Job 공개시점
- 제공 ETA 입력을 이용한 도착 전 준비 가능시간 계산
- priority rule과 tie-breaker
- action mask와 SMDP 할인
- queue-area·Reward·KPI 집계
- 익명화·시간대 변환·schema validation

### 1.2 Invariant Test

- 컨테이너 위치의 유일성과 tier 연속성
- YC·Job의 단일 할당
- 서비스영역, Hold, 슬롯 적합성
- 복수 YC 비통과·안전거리
- 이벤트 시각의 단조 증가
- 미공개 작업의 후보·관측값 유출 방지
- 안전제약 위반 0

### 1.3 Integration Test

- 원천자료 → 전처리 → simulator reset
- 한 작업의 공개 → 선택 → 이동 → 완료 → KPI 기록
- 사전 재조작 후 원작업 완료
- 장비고장·복구와 미완료 backlog
- Exp-1~3 정보필터 차이
- recorder → replay UI의 객체·KPI 일치
- Baseline/RL manifest 호환성 검사

### 1.4 Regression Test

고정된 소형 시나리오와 seed에 대해 다음을 저장한다.

- event 순서와 최종 yard-state hash
- 정책별 총비용·KPI 허용오차
- action·mask trace
- replay hash와 최종 UI KPI
- 제약 위반 수 0

모델·의존성·설정 변경 시 golden result와 비교한다. 부동소수점 결과는 합리적 tolerance를 사용하되 작업 선택순서의 변화는 별도 검토한다.

### 1.5 Simulator Validation

실제 로그를 학습하지 않은 validation 일자에 대해 비교한다.

- 작업유형·거리구간별 서비스시간 분포
- YC 이동거리와 idle/handling 비율
- 시간대별 처리량과 queue 길이
- 재조작률과 장치율별 변화
- 본선·내부작업 대기
- 고장 후 복구패턴

평균만 맞추지 않고 분위수, 분포형태, 상관관계를 확인한다. 시뮬레이터가 운영패턴을 충분히 재현하지 못하면 RL 성능평가를 진행하지 않는다.

## 2. 설정파일 예시

```yaml
experiment:
  id: exp3c
  information_level: pre_advice
  use_provided_eta: true
  eta_prediction_model: null
  control_scope: plus_pre_rehandle
  episode_unit: operating_day

terminal:
  profile: configs/terminals/target_terminal.yaml
  timezone: Asia/Seoul

environment:
  decision_horizon_sec: 1800
  max_candidates: 20
  drain_window_sec: 7200
  smdp_discount_ref_sec: 60

policy:
  type: tabular_q_learning
  actions:
    - FIFO
    - LONGEST_WAIT
    - NEAREST_JOB
    - MIN_REHANDLE
    - VESSEL_PRIORITY
    - EARLIEST_PROVIDED_ARRIVAL
    - PRE_REHANDLE
    - SAME_BAY_BATCH
    - WAIT_YIELD

evaluation:
  paired_seeds: [11, 17, 23, 31, 43]
  replay_recording: decision_and_event
  metrics:
    - queue_area
    - mean_wait
    - p95_wait
    - empty_travel
    - rehandles
    - throughput
    - unfinished_backlog
    - vessel_internal_delay
```

`use_provided_eta`는 제공값을 정책에 공개한다는 뜻이다. ETA 모델 학습이나 정확도 평가 옵션은 구현하지 않으며 `eta_prediction_model`은 명시적으로 `null`이다.

## 3. 핵심 처리 의사코드

```python
def step(selected_rule):
    assert action_mask[selected_rule]

    candidates = candidate_generator.visible_and_feasible(state)
    job = rule_executor.select(selected_rule, candidates, state)
    constraints.assert_dispatchable(job, state)

    simulator.reserve(job)
    simulator.advance_to_next_decision()

    interval_cost = reward_calculator.calculate(last_time, now, state)
    recorder.record_events_and_snapshot(state)

    observation = observation_encoder.encode(state)
    next_mask = action_masker.build(state)
    return observation, -interval_cost, is_done(state), {
        "action_mask": next_mask,
        "elapsed_time": now - last_time,
        "kpis": metrics.current(),
    }
```

## 4. 단계별 구현 로드맵

### Phase 0. 범위·자료 확정

- 대상 터미널·블록·YC 확정
- TerminalProfile의 사실값·가정값 구분
- 현행 dispatch rule과 작업분류 확인
- 데이터 사전·익명화·반출절차 합의

완료조건: 실제 구조를 반영한 profile과 데이터 샘플이 검토됨.

### Phase 1. 도메인·전처리

- Enum·객체·schema·validator
- 이벤트 정렬, 좌표·시간 표준화
- 실제 도착흐름과 사전정보 연결
- 날짜별 train/validation/test 분리

완료조건: 샘플 운영일이 오류·제외 보고서와 함께 로딩됨.

### Phase 2. 단일 YC 시뮬레이터

- EventQueue, YardState, Stack, CraneResource
- 이동·서비스시간과 작업 transition
- Hard Constraint와 invariant test
- 실제 로그 기반 validation

완료조건: 고정 시나리오 재현성과 validation 허용기준 충족.

### Phase 3. Baseline

- 현행 규칙, FIFO, LONGEST_WAIT, NEAREST_JOB, MIN_REHANDLE
- 공통 KPI·Reward·runner
- weighted heuristic 튜닝

완료조건: 모든 Baseline이 같은 사건열에서 실행되고 결과가 설명 가능함.

### Phase 4. Tabular Q-learning PoC

- 집계 State, CandidateGenerator, ActionMask
- priority-rule selector와 SMDP update
- validation checkpoint 선택

완료조건: 학습 재현성, 미방문 상태 fallback, 제약 위반 0.

### Phase 5. Exp-1~3

- 정보 공개필터 구현
- sequence-only의 순수 정보시점 비교
- 포지셔닝·사전 재조작의 단계별 ablation
- 운영부하별 paired evaluation

완료조건: 제공 ETA는 Exp-3 외생입력으로만 사용되고 ETA 정확도 실험이 없음.

### Phase 6. 검증 UI MVP

- replay recorder와 데이터 계약
- 단일 시나리오 재생, stack 상세, 정책 설명
- Baseline/RL 동기 비교
- UI-data regression test

완료조건: 같은 scenario·seed의 두 정책을 동기 비교하고 모든 화면값이 로그와 일치함.

### Phase 7. 함수근사 정책

- 필요성 판단 후 Masked DQN/PPO 구현
- 동일 reward·constraint·실험조건 사용
- Tabular·강한 heuristic 대비 추가효과 평가

완료조건: 복잡도 증가를 정당화할 일관된 out-of-sample 개선이 있음.

### Phase 8. 다중 YC Exp-4

- 실제 공통 서비스영역과 비통과·안전거리
- 중앙집중형 공동할당
- 간섭·양보·교착 테스트 및 UI 표시
- 필요 시에만 Multi-Agent 확장

완료조건: 실제 구조를 반영하고 간섭 제약 위반 0.

### Phase 9. 최종 평가

- 봉인된 test 일자 반복평가
- paired 통계·부하별 결과·ablation
- guardrail·실패사례·한계 보고
- 자료가 있을 때 사후 탄소 기대효과 산출

완료조건: 코드·설정·자료버전·seed로 결과를 재현할 수 있음.

- [ ] 대상 터미널·블록·YC가 확정됐는가?
- [ ] 공통 서비스영역·비통과·안전거리의 근거가 있는가?
- [ ] 현행 작업지시 규칙을 운영사가 확인했는가?
- [ ] 실제 차량 도착, 작업, YC, 장치상태 자료가 연결되는가?
- [ ] 사전 반출입정보와 제공 ETA 필드·제공시점이 정의됐는가?
- [ ] 가정값과 실측값이 설정에서 구분되는가?

### 실험·평가

- [ ] Exp-1~3의 실제 도착흐름과 초기상태가 동일한가?
- [ ] 정보 공개시점과 제어범위를 별도 요인으로 두었는가?
- [ ] Reward scale과 weight가 test 전에 고정되는가?
- [ ] 평균뿐 아니라 P95·backlog·본선·내부작업을 평가하는가?
- [ ] 탄소가 정책목표가 아니라 사후 기대효과로만 남아 있는가?

### 시각화

- [ ] UI가 읽기 전용 replay 구조인가?
- [ ] 정책 입력·후보·mask·선택이 화면에서 추적되는가?
- [ ] Baseline/RL 비교 run의 hash 호환성을 검사하는가?
- [ ] stack·YC 경로·안전거리의 물리오류를 확인할 수 있는가?
- [ ] UI의 KPI가 원 결과파일과 일치하는가?
