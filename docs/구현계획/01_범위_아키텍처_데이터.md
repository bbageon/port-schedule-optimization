# 01. 범위·아키텍처·데이터

[← 상위 구현계획서](../../부산항_야드크레인_강화학습_구현계획서.md)

## 1. 구현 범위

### 1.1 1차 PoC

- 선정한 부산항 터미널의 야드블록 1개, YC 1기
- 외부트럭 반입·반출작업
- 본선·내부 이송작업은 실제 작업으로 재현하고 서비스 수준을 보호
- 작업순서 선택, 사전 포지셔닝, 조건부 사전 재조작
- 사전 반출입정보와 부산항 제공 ETA를 외부 입력으로 사용
- 평균 대기시간과 P90/P95 대기시간을 주지표로 평가

### 1.2 2차 확장

- 실제 작업교환이 가능한 공통 서비스영역의 YC 2기 이상
- 비통과, 최소 안전거리, 인계지점 점유, 작업영역 중복
- 크레인별 작업량 균형, 대기·양보, 공동작업 할당
- 중앙집중형 스케줄러와 필요 시 협조형 Multi-Agent 정책 비교

### 1.3 제외 범위

- ETA 생성·예측모델 및 예측 정확도 평가
- 크레인의 물리적 레일 재배치와 서로 다른 터미널 간 이동
- 선박 전체 적부계획 및 안벽크레인 전체 스케줄 최적화
- 실제 장비에 자동 제어명령 전송
- 탄소배출량을 직접 최소화하는 정책

## 2. 터미널 프로파일

부산항 터미널별 장치장과 장비 구조가 다르므로 코드는 부산항 전체를 하나의 고정 구조로 가정하지 않는다. `TerminalProfile` 설정파일이 실제 블록 구조와 운영규칙을 주입한다.

| 그룹 | 필수 필드 |
|---|---|
| 기본정보 | terminal_id, profile_date, timezone, block_ids |
| 블록구조 | bay_count, row_count, tier_count, slot_coordinates |
| 크레인 | crane_id, crane_type, service_bay_min/max, can_cross, safety_distance |
| 이동성능 | gantry_speed, trolley_speed, loaded/empty_hoist_speed, acceleration |
| 작업시간 | lock/unlock_time, truck_positioning_time, handover_time |
| 차선·인계 | transfer_points, lane_capacity, allowed_job_types |
| 안전 | forbidden_zones, simultaneous_work_rules, wind_limit |
| 운영 | long_wait_sla, vessel_priority_rule, decision_horizon |

설정값의 근거 우선순위는 다음과 같다.

1. PLC·TOS·운영로그에서 산출한 실측 분포
2. 운영사 장비제원과 현장 운영규칙
3. 제조사 제원
4. 선행연구 파라미터
5. 임시 PoC 가정값

임시값은 `assumed: true`, 출처, 적용일을 기록한다. 실제 운영사 자료로 확인하기 전까지 `can_cross`, 안전거리, 해측·육측 역할, 서비스영역을 확정값처럼 쓰지 않는다.

### 2.1 실제 레인·인계자원 표현

레인은 하나의 `L1/L2` 목록으로 단순화하지 않는다. 실제 터미널의 차량 이동 도로와
YC가 차량을 처리하는 정차 위치는 서로 다른 자원이다.

| 객체 | 의미 | 필수 속성 |
|---|---|---|
| RoadSegment | 차량이 이동하는 도로 구간 | from/to, 방향, 허용차종, 용량, 이동시간 |
| TransferPoint | YC 아래 차량 정차·인계 위치 | block, side, 허용차종, 동시용량, 제어방식 |
| CraneRole | 크레인의 물리적 담당영역 | service side, 허용작업, 통과·안전거리 |

부산항 공개근거 프로파일은 [YR-002-a](../../.claude/docs/YR-002-부산항-야드동선-프로파일.md)를 따른다.
DGT는 수직형 블록의 육측 `LSTP`(외부트럭)와 해측 `WSTP`(AGV)를 hard separation하고,
HJNC는 수평형 twin-cantilever 양측 인계를 표현한다. 공개되지 않은 주도로 일방통행 edge와
인계점 동시용량은 `null + assumed`로 유지하고 운영사 도면 전까지 hard-code하지 않는다.

## 3. 시스템 아키텍처

```text
운영자료·TerminalProfile
  ├─ 컨테이너·슬롯 스냅샷
  ├─ 차량·작업 이벤트
  ├─ 제공 ETA 이력
  ├─ YC 이동·작업 로그
  └─ 장비상태 로그
            ↓
정제·검증 계층
  ├─ ID 익명화
  ├─ 시간·좌표 표준화
  ├─ 작업흐름 분류
  └─ 날짜 기준 train/validation/test 분리
            ↓
이산사건 시뮬레이터
            ↓
공통 RL 환경 + 정책
            ↓
실험 runner
  ├─ KPI·통계·사후 탄소평가
  └─ snapshot/event log → 읽기 전용 UI
```

시뮬레이터, 정책, UI는 분리한다. 정책은 표준 관측값만 받고, UI는 완료된 run의 snapshot과 event log만 읽는다. 따라서 UI의 재생·필터·속도 조정이 시뮬레이션 결과를 바꾸지 않는다.

## 5. 도메인 모델

### 5.1 주요 구분

```python
JobFlow = {
    GATE_IN, GATE_OUT,
    VESSEL_LOAD, VESSEL_DISCHARGE,
    TRANSSHIPMENT, REHANDLE
}
JobStatus = {PLANNED, RELEASED, WAITING, ASSIGNED, RUNNING, DONE, CANCELLED}
CraneStatus = {IDLE, MOVING, HANDLING, BLOCKED, DOWN}
LoadStatus = {FULL, EMPTY}
ContainerSize = {FT20, FT40, FT45}
```

반입·반출과 공·적 상태는 서로 다른 축이므로 하나의 Enum으로 합치지 않는다.

### 5.2 핵심 객체

`ContainerState`

| 필드 | 설명 |
|---|---|
| container_id_hash | 비가역 익명화 ID |
| size, load_status, special_flags | 규격·적공·reefer/DG/OOG 등 |
| block, bay, row, tier | 현재 장치위치 |
| work_available | 세관·검사·Hold를 반영한 작업가능 여부 |
| blockers | 목표 위 컨테이너 목록 |

`Job`

| 필드 | 설명 |
|---|---|
| job_id, flow, status | 식별자·작업종류·상태 |
| release_time | 현재 정보수준에서 정책에 공개되는 최초시각 |
| actual_arrival | 실제 차량 또는 작업 도착시각 |
| provided_eta | Exp-3에서 제공되는 외부 도착예정시각 |
| eta_provided_at | 해당 값이 시스템에 전달된 시각 |
| deadline, priority_class | 본선·운영상 마감과 우선순위 |
| target_container, target_slot | 대상 컨테이너와 목적 슬롯 |
| reachable_cranes | 작업 가능한 YC 목록 |

`CraneState`는 현재 Bay/Row, spreader 높이, 가용시각, 작업, 상태, 서비스영역, 동일 레일상의 순서를 가진다. `TruckState`는 gate 진입, block 도착, 서비스 시작·완료시각을 가진다.

`provided_eta`는 예측모델의 결과를 평가하기 위한 필드가 아니다. Exp-3에서 미래 작업을 공개하고 준비 가능한 시간창을 계산하기 위한 외생 입력이다.

## 6. 입력 데이터

### 6.1 최소 데이터셋

| 데이터셋 | 필수 필드 | 용도 |
|---|---|---|
| job_events | job_id, flow, actual_arrival, block, bay | 작업발생과 실제 도착 재현 |
| pre_advice | job_id, provided_eta, provided_at | Exp-3 정보 공개와 준비시간 계산 |
| crane_tasks | crane_id, job_id, start/end, from/to_bay | 처리·이동시간 보정 |
| container_snapshot | container_id, block/bay/row/tier, attributes | 초기 장치상태 |
| rehandle_log | source/destination_slot, start/end | 재조작 전이 보정 |
| equipment_status | equipment_id, down/recovery_time | 장비가용성 재현 |

Exp-1·2는 같은 `job_events`를 사용하되 정책 공개시점만 다르게 만든다. Exp-3은 `pre_advice`를 추가로 공개한다. 제공 ETA가 없는 자료는 Exp-3 핵심 표본에서 제외사유와 비율을 보고하며, 임의의 예측값을 만들어 채우지 않는다.

### 6.2 권장 데이터

- Gantry·Trolley·Hoist 축별 PLC 로그
- 작업차선·인계지점 점유이력
- Hold·검사·작업가능 상태
- 내부 이송장비 도착·대기·서비스 이력
- 외부트럭 공회전 및 YC 연료·전력 계측값

### 6.3 표준화·품질 규칙

- 원본 시각은 `Asia/Seoul`로 해석하고 내부는 UTC epoch second로 저장
- 위치는 `block-bay-row-tier`와 meter 좌표를 함께 유지
- 같은 timestamp는 고정 event 우선순위와 ID 순으로 처리
- 학습·검증·평가는 작업 단위가 아닌 날짜 단위로 분리
- 오류자료는 자동 삭제하지 않고 flag, 제외근거, 비율을 기록

```text
missing_pre_advice
invalid_slot
duplicate_event
negative_duration
unmatched_job
out_of_service_range
hold_violation
```

제공 ETA의 산출방식과 정확도는 분석하지 않는다. 입력자료 자체의 누락 여부만 적재 검증에서 확인한다.
