# 04. 시각화·검증 UI

[← 상위 구현계획서](../../부산항_야드크레인_강화학습_구현계획서.md)

## 1. 목적과 범위

UI의 1차 목적은 발표용 애니메이션이 아니라 **시뮬레이터와 정책의 검증**이다. 다음 질문에 답할 수 있어야 한다.

- 선택된 YC 작업이 현재 공개정보와 제약조건에서 타당한가?
- 사전 포지셔닝·재조작이 차량 도착 전에 실제로 끝났는가?
- 대기 감소가 불필요한 YC 이동이나 재조작 증가로 만들어진 것은 아닌가?
- 복수 YC가 겹치거나 통과하거나 교착상태에 빠지지 않는가?
- Baseline과 RL의 차이가 언제, 어떤 정책결정에서 생겼는가?
- 제공 ETA가 미래 작업을 공개하고 준비시간을 계산하는 용도로만 사용됐는가?

UI는 읽기 전용이다. 정책 실행 중 상태를 수정하거나 사람이 작업을 재배정하는 기능은 1차 범위에 포함하지 않는다. 3D 모델도 필요하지 않으며, 2D 평면도와 선택한 Bay의 stack 상세화면으로 충분하다.

## 2. 기술 구조

### 2.1 MVP 기술

- 시뮬레이터·정책: Python
- UI: Streamlit
- 평면도·차트: Plotly
- 결과 저장: Parquet + JSON metadata
- 실행: 완료된 run을 선택해 replay

```text
Simulator / Policy
        │
        ├─ metrics.parquet
        ├─ snapshots.parquet
        ├─ events.parquet
        ├─ decisions.parquet
        └─ run_manifest.json
                     ↓
              ReplayRepository
                     ↓
        Streamlit + Plotly UI
```

실시간 시뮬레이션과 UI를 직접 결합하지 않는다. runner가 표준 replay 자료를 생성하고 UI가 이를 읽는다. 객체 수·갱신속도가 Streamlit 한계를 실제로 넘을 때만 FastAPI + React + PixiJS/Konva + WebSocket을 검토한다.

### 2.2 코드 구성

```text
src/yard_rl/
├─ experiments/recorder.py       # snapshot/event/decision 기록
└─ ui/
   ├─ app.py                     # 페이지·run 선택
   ├─ replay.py                  # 시간축, 동기화, 상태 조회
   └─ components/
      ├─ yard_map.py             # 블록·YC·작업 표시
      ├─ policy_panel.py         # 후보·rule·mask·Q-value
      ├─ kpi_panel.py            # 현재·누적 KPI
      ├─ event_log.py            # 사건 필터·검색
      └─ compare_view.py         # Baseline/RL 동기 비교
```

## 3. Replay 데이터 계약

### 3.1 run manifest

```json
{
  "run_id": "20260712_exp3c_qlearning_seed17",
  "scenario_id": "test_2026_06_18_peak",
  "policy_id": "qlearning_v12",
  "experiment_id": "exp3c",
  "seed": 17,
  "terminal_profile_hash": "...",
  "event_stream_hash": "...",
  "initial_state_hash": "...",
  "model_hash": "...",
  "config_hash": "...",
  "time_start": 1781708400,
  "time_end": 1781794800
}
```

Baseline/RL 비교 run은 `scenario_id`, `seed`, `event_stream_hash`, `initial_state_hash`가 같아야 한다. 다르면 UI가 비교를 막고 차이 필드를 보여준다.

### 3.2 snapshots

snapshot은 의사결정 직전·직후와 주요 이벤트 직후에 기록한다.

| 필드 | 설명 |
|---|---|
| run_id, sim_time, snapshot_seq | run과 시간축 키 |
| crane_states | YC 위치·상태·현재작업·서비스영역 |
| truck_states | 블록대기·서비스 중 차량 |
| stack_summary | Bay/Row별 적층 높이와 선택 컨테이너 위치 |
| disclosed_jobs | 현재 정책이 볼 수 있는 작업만 포함 |
| hidden_job_count | UI 검증용 개수만; 상세내용은 노출하지 않음 |
| transfer_point_states | 인계지점·차선 점유 |
| current_kpis | 해당 시각 누적 KPI |

Exp-1·2 화면에서 Exp-3 사전정보가 우연히 보이지 않도록 recorder가 아니라 정책의 `InformationFilter` 결과를 저장한다.

### 3.3 decisions

| 필드 | 설명 |
|---|---|
| decision_id, sim_time, crane_id | 의사결정 식별 |
| state_key / observation_digest | 정책 입력 추적 |
| candidate_job_ids | 검토 후보 |
| action_mask | rule별 허용 여부 |
| action_scores | Q-value 또는 policy probability |
| selected_rule | 선택한 priority rule |
| selected_job_id | rule이 결정한 실제 작업 |
| tie_break_trace | 동률 해소 근거 |
| constraint_checks | 최종 제약검사 결과 |
| reason_codes | 사용자용 한 줄 설명 구성요소 |

`reason_codes` 예시는 `LONGEST_WAIT`, `ETA_WITHIN_HORIZON`, `ZERO_BLOCKER`, `NEAREST_CRANE`, `VESSEL_GUARDRAIL`, `MASKED_NO_SAFE_SLOT`이다.

### 3.4 events

| 필드 | 설명 |
|---|---|
| event_id, sim_time, type | 사건 키 |
| entity_type, entity_id | 대상 객체 |
| before_state, after_state | 상태변화 요약 |
| related_job_id, crane_id | 관련 작업·YC |
| duration, distance | 해당 시 포함 |
| severity | info / warning / error |

원본 컨테이너·차량 식별자는 UI 자료에도 저장하지 않는다.

## 4. 화면 1: 단일 시나리오 Replay

### 4.1 레이아웃

```text
┌──────────────────────────────────────────────────────────────┐
│ Run / Exp / Policy / Seed       재생  일시정지  Step  속도   │
├───────────────────────────────────┬──────────────────────────┤
│                                   │ 정책 결정                │
│  Yard 2D 평면도                   │ - 선택 rule / 작업       │
│  - Bay·Row·차선·인계지점          │ - 후보와 점수            │
│  - YC 위치·경로·서비스영역        │ - action mask 사유        │
│  - 차량·공개작업·컨테이너 이동    │ - 제약검사               │
│                                   ├──────────────────────────┤
│                                   │ 현재·누적 KPI            │
├───────────────────────────────────┴──────────────────────────┤
│ 시간축 / 주요 이벤트 marker / event log                    │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 조작

- Play/Pause, 이전·다음 의사결정 Step
- 0.25x, 0.5x, 1x, 2x, 5x, 10x 재생
- 시간축 이동 및 `long wait`, `rehandle`, `interference`, `constraint warning` marker 점프
- 특정 YC, Job, 차량, Bay 필터
- stack 클릭 시 Bay-Row-Tier 상세 popup
- 선택 작업의 원인과 후보 비교 표시

### 4.3 색상 규칙

| 색 | 의미 |
|---|---|
| 회색 | 정책에 아직 공개되지 않은 작업의 집계영역; 개별 객체는 숨김 |
| 파랑 | 블록에서 대기 중인 외부트럭 작업 |
| 하늘색 | Exp-3에서 제공 ETA로 공개된 미래작업 |
| 주황 | 사전 포지셔닝·사전 재조작 |
| 초록 | 완료작업 |
| 빨강 | SLA 초과·간섭위험·오류 |
| 보라 | 본선·내부 이송 연계작업 |

색만으로 상태를 구분하지 않고 icon·label·pattern을 병행한다.

### 4.4 정책 설명 패널

패널은 다음 문장을 자동 구성할 수 있어야 한다.

> 10:31:20에 YC-02는 `PRE_REHANDLE`을 선택했다. Job-184의 제공 도착예정시각까지 11분 40초가 남았고 blocker 1개와 안전한 목적 슬롯이 확인됐다. 본선 마감 및 현재 SLA 작업은 guardrail을 만족했다.

Q-value가 절대적인 인과설명인 것처럼 표현하지 않는다. “정책이 사용한 입력·후보·점수·rule·제약검사”를 사실대로 보여준다.

## 5. 화면 2: Baseline vs RL 동기 비교

두 정책의 run을 동일한 절대 시각축에 놓고 함께 재생한다.

```text
┌────────────────────────────┬────────────────────────────┐
│ Baseline                   │ RL                         │
│ Yard map + current action  │ Yard map + current action  │
├────────────────────────────┴────────────────────────────┤
│ Δ 대기 / Δ 이동 / Δ 재조작 / Δ 처리량 / guardrail     │
│ 공통 시간축, 공통 외부사건, 정책별 결정 marker         │
└─────────────────────────────────────────────────────────┘
```

### 5.1 동기화 기준

- 외부사건은 같은 실제 `sim_time`에 표시
- 정책 의사결정 시점은 서로 다를 수 있으므로 각 화면의 마지막 유효 snapshot을 유지
- 한쪽 정책이 작업을 빨리 끝내도 상대 정책의 시계를 억지로 앞당기지 않음
- 같은 event stream이지만 정책결과로 생기는 내부 이벤트는 각 run의 색으로 구분
- 비교 가능한 run인지 manifest hash로 먼저 검증

### 5.2 비교 지표

- 현재 대기차량 수와 누적 queue-area 차이
- 평균·P95 대기시간 차이
- 적재·비적재 이동거리 차이
- 재조작·계획변경 차이
- 완료 처리량과 미처리 backlog 차이
- 본선·내부작업 지연 및 안전 guardrail

대기시간만 녹색으로 강조하지 않는다. 대기가 감소했지만 이동·재조작·보호지표가 악화되면 trade-off 경고를 함께 표시한다.

## 6. Stack 상세와 물리 검증

Bay를 선택하면 Row-Tier 단면을 표시한다.

- 컨테이너 크기·적공·특수속성
- 목표 컨테이너와 blocker 순서
- 사전 재조작 원위치·목적 슬롯
- 현재 top tier와 슬롯 적합성
- Hold·금지 슬롯
- 선택 YC의 도달 가능 여부

작업 전후 stack을 나란히 표시해 공중에 뜬 컨테이너, 중복 위치, 잘못된 목적 슬롯, blocker 미제거 같은 오류를 찾는다.

다중 YC 화면은 서비스영역, 안전거리 band, 예상 이동경로, 공통 인계지점 점유를 겹쳐 보여준다. 두 YC의 경로가 안전거리 안으로 들어가면 시뮬레이터가 작업을 막아야 하며, UI는 단순 경고만 하고 통과시키면 안 된다.

## 7. KPI와 이벤트 로그

### 7.1 현재 KPI

- 현재 대기차량, 최장대기, SLA 초과
- YC별 상태와 남은 작업시간
- 현재까지 이동거리·재조작·처리량
- 본선·내부작업 마감 임박

### 7.2 종료 KPI

- 평균·중앙값·P90·P95·최대 대기
- queue-area와 종료 backlog
- 적재/비적재 이동거리
- 재조작, idle/blocked/interference 비율
- 본선·내부작업 지연
- 안전제약 위반

### 7.3 로그 필터

`decision`, `truck`, `crane`, `rehandle`, `positioning`, `interference`, `equipment`, `warning/error`로 필터한다. 특정 행을 누르면 시간축과 yard map이 그 시점으로 이동한다.

## 8. UI 검증 테스트

### 8.1 데이터 일치

- 같은 run·같은 시각의 UI 객체 수가 snapshot과 일치
- UI 누적 KPI가 최종 `metrics.parquet`와 일치
- decision panel의 mask·선택 rule·job이 정책 로그와 일치
- 컨테이너 이동 전후 위치가 event log와 일치
- Exp-1·2 화면에서 제공 ETA·미래작업 상세가 노출되지 않음

### 8.2 재현성과 비교

- 같은 seed·설정의 replay hash가 동일
- 서로 다른 event/initial-state hash를 가진 run은 비교 불가
- Baseline/RL의 공통 외부사건 timestamp가 일치
- 재생속도·필터·화면 이동이 KPI나 run 자료를 변경하지 않음

### 8.3 물리·안전

- 동일 슬롯의 컨테이너 중복 표시 없음
- tier 불연속·공중 적재 없음
- YC 서비스영역 밖 작업 표시 없음
- 비통과 YC 순서 역전 없음
- 안전거리 위반 경로와 불법 사전 재조작 없음

### 8.4 성능

MVP 기준으로 운영일 1일, 블록 1개, YC 1~2기의 replay를 일반 개발 PC에서 3초 이내 로딩하고, Step 이동은 300ms 이내를 목표로 한다. 모든 이벤트를 매 frame 다시 그리지 않고 snapshot index와 시간구간 캐시를 사용한다.

## 9. UI 구현 순서와 완료기준

### UI-1. Recorder

- manifest, snapshot, event, decision 스키마 구현
- 단일 run의 최종 KPI 일치 검증

### UI-2. 단일 Replay

- Yard 2D, Play/Pause/Step, 시간축, KPI, event log
- YC·차량·컨테이너 이동과 stack 상세 표시

### UI-3. 정책 설명

- 후보, Q-value/probability, action mask, 선택 rule·job, reason code
- Exp별 정보 노출 검증

### UI-4. 동기 비교

- manifest 호환성 검사
- Baseline/RL 좌우 동기 재생과 KPI 차이

### UI-5. 다중 YC 검증

- 서비스영역, 안전거리, 경로, 양보·간섭 표시

UI MVP 완료조건은 다음과 같다.

1. 한 운영일의 단일 정책 실행을 처음부터 끝까지 재생할 수 있다.
2. 임의의 의사결정에서 정책 입력·후보·mask·선택결과를 확인할 수 있다.
3. 동일 시나리오의 Baseline과 RL을 같은 시각축에서 비교할 수 있다.
4. UI가 시뮬레이션 결과를 변경하지 않음이 regression test로 보장된다.
5. 물리·안전 불변조건의 위반이 화면과 자동검사 모두에서 0이다.
