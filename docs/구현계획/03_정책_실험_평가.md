# 03. 정책·실험·평가

[← 상위 구현계획서](../../부산항_야드크레인_강화학습_구현계획서.md)

## 1. 비교 정책

### 1.1 필수 Baseline

| 정책 | 목적 |
|---|---|
| 현행 작업지시 규칙 | 실제 운영 기준선; 운영사 로직 확인이 필요 |
| FIFO | 작업 공개·도착순 기준선 |
| LONGEST_WAIT | 대기시간 직접 우선 기준선 |
| NEAREST_JOB | YC 이동 최소화 휴리스틱 |
| MIN_REHANDLE | 재조작 최소화 휴리스틱 |
| Weighted heuristic | 운영 우선순위·대기·거리·재조작을 합친 강한 기준선 |

현행 로직이 불명확하면 임의로 “TOS 최적화”라 부르지 않는다. 운영사가 확인한 dispatch rule과 재현 차이를 문서화한다.

### 1.2 동적 후보 평가형 Deep Q

Tabular·rule 선택 정책은 PoC 증거로 보존하고 신규 함수근사 정책은 실행 가능한 Job을
직접 평가한다.

\[
Q_{cost}(g,c_i,x_j)\rightarrow\mathbb{R},\qquad
a_i^*=\arg\min_{j:\,mask_{ij}=1}Q_{cost}(g,c_i,x_j)
\]

- Candidate DQN: network·mask·replay 기준 동작 확인
- **Candidate Double DQN**: online 선택과 target 평가를 분리한 기본 학습기
- Dueling Candidate Double DQN: masked advantage 분해의 추가효과 비교
- 학습: epsilon-greedy + dynamic candidate mask + replay/target network
- 평가: 탐색 0, 결정론적 tie-break, 고정 scenario manifest
- 저장: network·target/scaler·feature schema·cost scale·seed·config hash

가변 후보를 고정 action ID로 매핑하지 않는다. 후보는 같은 Q-network로 batch 평가하며
GPU는 학습에 권장하되 CPU fallback과 CPU 운영 추론을 보장한다. 상세 구현은
[06](06_동적후보_Deep_Q_다중YC.md)을 따른다.

### 1.3 다중 YC

다중 YC의 첫 기준선은 YC-Job 후보비용을 이용한 중앙 공동 matching이다. 동일 Job 중복,
레인 점유, 비통과·안전거리를 만족하는 joint action만 비교한다. 로컬 Candidate Double
DQN과 중앙 resolver가 검증된 뒤 같은 로컬 utility를 QMIX mixer에 연결해 CTDE의
추가효과를 paired 비교한다. QMIX를 검증되지 않은 로컬 평가망과 동시에 도입하지 않는다.

## 2. 학습 파이프라인

### 2.1 데이터 분할

- 날짜 단위 train/validation/test 분리
- 인접 날짜의 동일 운영상황 누출 여부 점검
- test 기간은 정책·weight·bucket 결정 이후까지 봉인
- 같은 평가일과 seed를 모든 정책에 공통 사용

### 2.2 Episode

한 episode는 운영일 1일 또는 연속 운영구간이다. 초기 장치상태, 실제 차량 도착흐름, 본선·내부 작업, 장비고장 이벤트는 정책 간 동일하다.

종료조건은 운영구간 종료 후 지정된 drain window 종료다. 미처리 backlog는 종료비용과 KPI에 포함해 작업을 미루는 정책이 유리해지지 않도록 한다.

### 2.3 기록 항목

```text
run_id, policy_id, experiment_id, scenario_id, seed
episode_cost, queue_area, mean/p90/p95/max_wait
loaded/empty_travel, rehandles, throughput, unfinished_backlog
vessel/internal delay, interference_wait, constraint_violation
action_counts, masked_actions, state_visits, fallback_count
snapshot_uri, event_log_uri, model_version, config_hash
```

모델은 validation의 대기비용을 우선하되 처리량·본선·내부작업·안전 guardrail을 만족하는 checkpoint 중에서 선택한다.

## 3. 실험 매트릭스

### 3.1 핵심 실험

| 실험 | 정책 정보 | control_scope | 질문 |
|---|---|---|---|
| Baseline | 현행 공개정보 | 현행 범위 | 현재 운영의 기준 성능은? |
| Exp-1 | 블록 도착 이후 | sequence_only | 도착 후 작업순서 최적화 효과는? |
| Exp-2 | 게이트 진입 이후 | sequence_only | 정보가 조금 일찍 공개될 때의 효과는? |
| Exp-3A | 사전정보 + 제공 ETA | sequence_only | 순서 조정만으로 얻는 효과는? |
| Exp-3B | 사전정보 + 제공 ETA | plus_positioning | 사전 포지셔닝 추가효과는? |
| Exp-3C | 사전정보 + 제공 ETA | plus_pre_rehandle | 선재조작까지 허용한 추가효과는? |
| Exp-4 | Exp-3 정보 + 복수 YC | 단계별 동일 범위 | 협조·간섭회피 효과는? |

ETA는 Exp-3의 입력조건이지 독립적인 예측 성능 실험축이 아니다. ETA 정확도와 오차에 관한 시나리오는 핵심 실험에서 제외한다.

### 3.2 운영부하 시나리오

| 요인 | 수준 예시 |
|---|---|
| 차량 도착부하 | 비혼잡 / 평시 / 피크 |
| 장치율 | 낮음 / 중간 / 높음 |
| 초기 YC 위치 | 근접 / 중립 / 불리 |
| 본선·내부작업 강도 | 낮음 / 보통 / 높음 |
| 장비가용성 | 정상 / 기록된 고장 재현 |
| YC 구성 | 단일 / 공통 서비스영역 복수 |

가능하면 인위적으로 분포를 만들기보다 실제 평가일을 부하구간으로 분류해 재생한다. 합성 stress scenario는 실제자료 범위 밖임을 명시한다.

### 3.3 Ablation

| Ablation | 제거·변경 | 확인 목적 |
|---|---|---|
| No positioning | 비적재 사전이동 제거 | 포지셔닝 기여 |
| No pre-rehandle | 선재조작 제거 | 선재조작 기여 |
| No movement term | 이동비용항 제거 | 대기 단일목표의 이동 악화 여부 |
| No tail term | P90/P95 보호항 제거 | 평균과 상위대기의 trade-off |
| No vessel guardrail | 본선 보호항 제거 | 외부트럭 편향 여부 |
| No action mask | 안전 mask 제외가 아니라 soft 제약 실험 | 설계상 사용 금지; 비교하지 않음 |

정보공개 효과는 Exp-1~3의 동일 `sequence_only` 비교로 확인한다. 제공 ETA 활용 자체를 제거한 결과는 Exp-2와 Exp-3A의 차이로 충분히 측정하며 ETA 정확도 실험으로 확대하지 않는다.

## 4. KPI

### 4.1 외부트럭

- 평균, 중앙값, P90, P95, 최대 블록 대기시간
- SLA 초과 차량 수·비율
- queue-area
- 반입·반출별 대기와 처리량
- 종료시점 미처리 차량 수와 누적대기

### 4.2 YC

- 적재·비적재 이동거리와 이동시간
- 총 에너지·연료 사용량: 계측자료가 있을 때만
- 재조작 횟수 및 컨테이너당 재조작
- idle, handling, blocked, interference waiting 비율
- 크레인별 작업량 불균형

### 4.3 터미널 운영

- 시간당 처리량
- 본선·내부 이송작업 대기와 deadline 위반
- 작업계획 변경·취소 횟수
- 안전제약 위반 수: 항상 0이어야 함
- 공통 인계지점·차선의 대기와 포화시간

평균 대기 감소만으로 정책을 채택하지 않는다. P95, 처리량, 본선·내부작업, backlog, 안전 guardrail을 동시에 확인한다.

## 5. 통계 분석

정책별로 같은 평가일·초기상태·사건·seed를 사용하는 paired design을 적용한다.

1. 정책 간 KPI 차이와 변화율 산출
2. 일자 또는 episode 단위 paired bootstrap 95% 신뢰구간
3. 분포가 비정규적이면 Wilcoxon signed-rank를 보조 사용
4. 효과크기와 원단위 차이를 p-value보다 우선 보고
5. 운영부하별 subgroup 결과 보고
6. 평균 개선과 P90/P95 악화 여부를 함께 표시

반복수는 사전 pilot의 분산으로 power 또는 신뢰구간 폭을 계산해 정한다. 임의의 고정 반복수만으로 충분하다고 가정하지 않는다.

### 5.1 잠정 합격조건

수치는 운영사와 pilot 후 확정하되 평가 논리는 다음처럼 고정한다.

- 평균 또는 queue-area가 Baseline보다 유의하게 감소
- P90/P95 대기가 허용범위 내에서 감소하거나 비열등
- 처리량·본선·내부작업 SLA가 사전 정의한 guardrail을 만족
- YC 이동·재조작이 악화되면 그 원인과 대기감소 대비 trade-off를 보고
- 안전제약 위반 0
- 서로 다른 평가일·seed에서 효과 방향이 지나치게 불안정하지 않음

## 6. 탄소중립 기대효과 사후평가

탄소는 RL의 State·Action·Reward·정책선택 기준에 포함하지 않는다. 다음 인과경로가 자료로 확인될 때만 운영단계 기대효과로 산출한다.

```text
차량 대기 감소 → 실제 엔진 공회전 감소 → 연료·배출 감소
YC 이동·재조작 감소 → 실제 전력·연료 사용 감소 → 배출 감소
```

### 6.1 실측자료가 있는 경우

\[
\Delta CO_{2e}^{truck}
=\sum_v \Delta t_v^{idle}\cdot EF_v^{idle}
\]

\[
\Delta CO_{2e}^{YC}
=\Delta E_{YC}\cdot EF_{electricity}
+\Delta F_{YC}\cdot EF_{fuel}
\]

- 차종·연료별 공회전계수와 적용연도를 기록
- 전력·연료 배출계수의 출처와 적용연도를 기록
- 평균값과 함께 불확실성 범위를 보고
- 차량 대기시간을 그대로 공회전시간으로 치환하지 않음

### 6.2 실측자료가 없는 경우

대기시간·이동거리·재조작 감소만 보고하고 “탄소 감축량”으로 단정하지 않는다. 표현은 `탄소 감축 잠재력`, `운영단계 탄소중립 기여 가능성`으로 제한한다.
