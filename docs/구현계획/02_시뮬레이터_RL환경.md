# 02. 시뮬레이터·RL 환경

[← 상위 구현계획서](../../부산항_야드크레인_강화학습_구현계획서.md)

## 1. 이벤트 기반 시뮬레이터

시뮬레이션은 1초 간격으로 전체 상태를 반복 계산하지 않고, 상태가 변하는 시점으로 시계를 이동하는 연속시간 이산사건 방식으로 구현한다.

### 1.1 주요 이벤트

| 이벤트 | 의미 |
|---|---|
| JOB_DISCLOSED | 정보수준에 따라 작업이 정책에 처음 공개됨 |
| TRUCK_GATE_IN | 차량이 게이트에 진입 |
| TRUCK_BLOCK_ARRIVAL | 차량이 대상 블록에 도착 |
| CRANE_AVAILABLE | YC가 다음 작업을 선택할 수 있음 |
| TASK_START / TASK_END | 이동·취급을 포함한 작업 시작·종료 |
| CONTAINER_MOVED | 컨테이너 위치가 변경됨 |
| EQUIPMENT_DOWN / RECOVERED | 장비 고장·복구 |
| DEADLINE_ALERT | 본선·내부작업 마감 임박 |

`provided_eta`는 별도의 예측 갱신 연구대상이 아니다. Exp-3의 `JOB_DISCLOSED` 시점에 작업 속성으로 주입하며, 해당 값으로 도착 전 준비 가능시간을 계산한다.

### 1.2 동시 이벤트 우선순위

같은 시각의 이벤트는 다음 순서로 처리하고 마지막에 의사결정을 한 번만 호출한다.

1. 안전·장비상태 변화
2. 수행 중 작업완료와 컨테이너 상태 갱신
3. 실제 차량·내부작업 도착
4. 작업정보 공개
5. deadline alert
6. YC 의사결정

### 1.3 상태 불변조건

- 컨테이너는 동시에 두 슬롯에 존재하지 않는다.
- 한 슬롯의 tier는 아래에서부터 연속으로 쌓인다.
- 하나의 YC는 동시에 하나의 작업만 수행한다.
- 하나의 작업은 동시에 두 YC에 배정되지 않는다.
- YC는 자신의 서비스영역 밖 작업을 수행하지 않는다.
- 비통과 구조에서는 동일 레일상의 YC 순서가 바뀌지 않는다.
- Hold·작업불가 컨테이너는 이동하지 않는다.
- `DONE` 작업은 다시 후보가 되지 않는다.

불변조건 위반은 큰 penalty로 학습시키지 않고 시뮬레이션 오류로 즉시 중단한다.

## 2. 이동·서비스시간 모델

YC의 예상 작업시간은 Gantry·Trolley·Hoist 이동, 가감속, lock/unlock, 차량 정위치, 인계 대기를 분리해 계산한다.

초기 PoC의 간소식은 다음과 같다.

\[
T_{move} = \max\left(\frac{|\Delta x|}{v_g}, \frac{|\Delta y|}{v_t}\right)
+ \frac{|\Delta z|}{v_h} + T_{acc/dec}
\]

\[
T_{service} = T_{move}^{empty} + T_{lock} + T_{move}^{loaded}
+ T_{unlock} + T_{positioning} + T_{interference}
\]

초기에는 장비로그에서 작업유형·거리구간별 empirical distribution을 추정해 표본추출한다. PLC 자료가 확보되면 축별 운동모델로 교체한다. 평균값 하나만 고정 사용하지 않으며 validation 일자에서 거리구간별 MAE, 분포, 처리량을 함께 검증한다.

다중 YC에서는 인접 YC 예상 궤적, 안전거리, 공통 인계지점 점유를 검사해 `T_interference`와 양보 이벤트를 산출한다.

## 3. 정보 공개와 제어범위

### 3.1 정보 공개시점

| 정보 | Exp-1 | Exp-2 | Exp-3 |
|---|:---:|:---:|:---:|
| 블록 도착 사실 | O | O | O |
| 게이트 진입 사실 | X | O | O |
| 사전 반출입정보 | X | X | O |
| 제공 ETA | X | X | O |
| 실제 도착시각 | 사후 기록 | 사후 기록 | 사후 기록 |

모든 실험은 같은 실제 도착흐름을 재생한다. 달라지는 것은 정책이 작업을 처음 볼 수 있는 시점이다. 제공 ETA를 직접 수정하거나 ETA 정확도를 점수화하지 않는다.

### 3.2 제어범위

| control_scope | 허용 행동 |
|---|---|
| sequence_only | 현재 실행 가능한 작업의 순서만 변경 |
| plus_positioning | 미래 공개작업 위치로 비적재 YC 이동 허용 |
| plus_pre_rehandle | 안전한 목적 슬롯이 있을 때 blocker 선처리 허용 |

정보 공개효과와 행동범위 효과를 분리하기 위해 Exp-1~3 비교에서는 `control_scope`를 동일하게 유지한다. 이후 별도 ablation에서 포지셔닝과 선재조작을 단계적으로 추가한다.

## 4. RL 환경 인터페이스

```python
observation, info = env.reset(seed=seed, options=scenario)
observation, reward, terminated, truncated, info = env.step(action)
```

`reset`은 TerminalProfile, 초기 장치상태, 실제 이벤트, 실험별 정보필터를 로딩하고 첫 의사결정 시점까지 진행한다.

`step`은 다음을 수행한다.

1. action mask 확인
2. 선택된 후보 action에서 실제 Job·준비행동 식별
3. 안전·도달영역·작업가능성 재검증
4. 작업 예약과 미래 이벤트 생성
5. 다음 의사결정 시점까지 시뮬레이션 진행
6. 경과구간 비용·KPI·snapshot 기록
7. 다음 관측값과 mask 반환

의사결정 사이 경과시간을 \(\tau\), 기준시간을 \(\Delta_{ref}\)라 하면 SMDP 할인은 다음과 같다.

\[
\gamma_{\tau}=\gamma_{ref}^{\tau/\Delta_{ref}}
\]

## 5. State와 후보 생성

### 5.1 Tabular PoC 집계상태

| Feature | Bucket 예시 |
|---|---|
| time_period | off_peak / normal / peak |
| crane_position_zone | 블록 내 4~6개 구간 |
| queue_length | 학습자료 분위수 기반 4구간 |
| oldest_wait | SLA·분위수 기반 4구간 |
| disclosed_future_count | 0 / low / medium / high |
| time_to_nearest_provided_eta | none / near / medium / far; Exp-3만 |
| candidate_distance | near / medium / far |
| min_blockers | 0 / 1 / 2 / 3+ |
| vessel_urgency | none / normal / urgent |
| crane_interference | Exp-4에서 none / possible / active |

`time_to_nearest_provided_eta`는 제공값의 활용을 위한 상태이며 예측오차 feature가 아니다. Exp-1·2에서는 `none`으로 고정한다.

### 5.2 함수근사 관측값

```text
observation = {
  global_features: float32[G],
  crane_features: float32[N_crane, F_crane],
  candidate_features: float32[K, F_job],
  candidate_valid_mask: bool[K],
  action_mask: bool[N_action]
}
```

후보작업은 현재 대기, 공개된 도착예정시각, release 여부, 작업유형, 거리, blocker 수, 호환 목적 슬롯, 예상 처리시간, deadline slack, 본선연계 여부, 도달가능 YC를 가진다. ETA 불확실성·예측오차는 넣지 않는다.

### 5.3 Top-K 후보

다음 후보를 먼저 포함한 후 priority score로 나머지를 채운다.

1. 최장대기 작업
2. SLA 초과 또는 임박 작업
3. 본선 마감 임박 작업
4. Exp-3에서 제공 ETA가 가까운 공개작업
5. idle YC에서 가까운 실행 가능 작업

같은 State·Event에서는 tie-breaker를 `SLA → deadline slack → 누적대기 → 예상처리시간 → job_id` 순으로 고정해 결정론적으로 동작시킨다.

## 6. 동적 후보 Action

정책 action은 priority rule 번호가 아니라 현재 실행 가능한 작업 자체다.

```text
Action ∈ {
  SERVE(job_id),
  PRE_REHANDLE(job_id),
  REPOSITION(target_bay),
  WAIT_UNTIL(next_public_event)
}
```

단일 YC PoC는 `SERVE(job_id)`만 사용한다. FIFO·LONGEST_WAIT·NEAREST_JOB·
MIN_REHANDLE·VESSEL_PRIORITY·SAME_BAY_BATCH는 action이 아니라 비교 휴리스틱이다.
함수근사는 고정 action head 대신 `[Global, YC, Candidate] → Q_cost` 공유망으로 후보별
예상 누적비용을 계산한다. 상세 계약은 [06](06_동적후보_Deep_Q_다중YC.md)을 따른다.

Action Masking은 아직 공개되지 않았거나 실행 불가능한 후보를 정책 점수 계산 전에
제거한다. `PRE_REHANDLE`은 Hold·목적 슬롯 부재·시간창 부족 시, `REPOSITION`은
도달영역·간섭 제약 위반 시, `WAIT_UNTIL`은 운영상 idle이 허용되지 않으면 mask한다.

모든 action이 mask되면 다음 외부 이벤트까지 자동 진행한다. feasible 후보가 있는데 모두
mask된 경우에는 조용히 진행하지 않고 후보·mask 계약 위반으로 중단한다.

### 6.1 사전 재조작 조건

- 대상 작업과 컨테이너가 정책에 이미 공개됨
- 컨테이너가 yard에 존재하고 작업 가능함
- 차량 도착 전 충분한 완료시간이 있음
- 합법적 목적 슬롯이 존재함
- 본선·현재 대기작업의 SLA를 침해하지 않음
- 재조작으로 생기는 미래 blocker 위험이 허용치 이내임

반입 예정 컨테이너에는 대상 컨테이너 선재조작을 적용하지 않고, 슬롯 계획과 YC 포지셔닝만 허용한다.

## 7. Reward

Exp-1~3에는 동일한 정규화 비용함수를 사용한다.

\[
\begin{aligned}
C_t={}&w_W\frac{A^{queue}_t}{S_W}+w_T\frac{A^{tail}_t}{S_T}\\
&+w_M\frac{D^{loaded}_t+(1+\eta_{empty})D^{empty}_t}{S_M}
+w_R\frac{N^{rehandle}_t}{S_R}\\
&+w_V\frac{T^{vessel/internal}_t}{S_V}
+w_C\frac{N^{planchange}_t}{S_C}
\end{aligned}
\]

\[
Reward_t=-C_t,\qquad
A^{queue}_t=\int_{time_t}^{time_{t+1}}Q_{external}(u)du
\]

- `queue-area`로 완료차량뿐 아니라 종료시점 미처리차량의 대기를 포함한다.
- 정규화 scale은 train 구간 Baseline의 robust statistic으로 산출한 뒤 고정한다.
- 1차 목표는 대기이지만 단일항 최적화로 이동·재조작·본선서비스가 악화되지 않도록 보호항을 둔다.
- weight는 validation에서 정하고 test 결과를 보고 변경하지 않는다.

Reward에 ETA 예측오차와 탄소배출량을 넣지 않는다. 제공 ETA를 이용한 잘못된 준비행동이 발생하면 그 결과로 생긴 실제 이동, 재조작, 대기, 계획변경 비용을 통해서만 반영된다.

## 8. Hard Constraint

### 8.1 YC 제약

- 서비스영역 밖 이동 금지
- 동일 레일 비통과와 최소 안전거리
- 공통 인계지점·차선 용량 초과 금지
- 장비고장·강풍 시 신규작업 금지
- spreader·컨테이너 규격 호환

### 8.2 컨테이너·슬롯 제약

- blocker가 남은 컨테이너 직접 반출 금지
- 빈 슬롯 또는 현재 top tier에만 적재
- tier·중량·규격·DG·reefer 제약 준수
- Hold·검사 상태 이동 금지
- 사전 재조작의 임시 목적 슬롯 예약

### 8.3 작업 제약

- 정책에 공개되지 않은 작업 선택 금지
- 차량 미도착 상태의 실제 상·하차 금지
- Exp-3의 공개 미래작업에는 포지셔닝·허용된 선재조작만 가능
- 본선·내부 이송 deadline hard rule이 있으면 외부트럭 작업보다 우선 적용

`ConstraintEngine`은 후보 생성, action mask, 실제 dispatch 직전의 세 단계에서 같은 규칙을 검사한다.
